# Design: Prefab Builder Migration — P7.4 to P7.7 (Approach A)

**Date:** 2026-04-27
**Scope:** Complete Phase 7 migration of all remaining `.tscn` prefabs to `U_EditorPrefabBuilder` scripts.
**Approach:** Minimal builder extension + imperative material setup in scripts (Approach A).

---

## 1. Problem Statement

Phase 7 of Cleanup V8 migrates hand-authored `.tscn` prefabs to programmatic builder scripts. Five static prefabs are already migrated. The remaining prefabs require builder features not yet implemented — specifically, attaching children to non-root nodes (inherited `Player_Body`) and overriding deeply nested inherited properties (hover ray `target_position`).

## 2. Builder Extensions (Minimal)

Two new methods added to `U_EditorPrefabBuilder`:

```gdscript
func add_child_to(parent_path: String, node: Node) -> U_EditorPrefabBuilder
func add_child_scene_to(parent_path: String, scene_path: String, child_name: String) -> U_EditorPrefabBuilder
```

Both resolve the parent via `_root.get_node(parent_path)`, guard nulls, add the child, and return `self` for fluent chaining.

**Rationale:** `override_property` already handles property overrides on any node path (e.g., `"Components/C_MovementComponent"`). These two methods fill the only remaining structural gap.

## 3. Material Setup Strategy

Materials on `Body_Mesh` CSG nodes are constructed imperatively inside builder scripts:

```gdscript
var body_mesh := CSGBox3D.new()
var mat := StandardMaterial3D.new()
mat.shading_mode = BaseMaterial3D.SHADING_MODE_UNSHADED
mat.diffuse_mode = BaseMaterial3D.DIFFUSE_LAMBERT_WRAP
mat.specular_mode = BaseMaterial3D.SPECULAR_TOON
mat.albedo_color = Color(0.24, 0.26, 0.29)
body_mesh.material = mat
builder.add_child_to("Player_Body", body_mesh)
```

This keeps the builder under its 200-line LOC cap and avoids adding a material DSL with only 4–5 consumers.

## 4. Prefab Categories and Migration Strategy

### 4a. Sub-Prefabs (1)
- `prefab_demo_npc_body` — `Node3D` root + `MeshInstance3D` (sphere) + `Sprite3D` (shadow blob). Trivial; no inheritance.

### 4b. Character Prefabs (4 demo)
- `prefab_woods_wolf`, `prefab_woods_rabbit`, `prefab_woods_builder`, `prefab_demo_npc`
- All inherit from `tmpl_character.tscn`.
- Pattern: `inherit_from()` → `set_entity_id()` → `set_tags()` → `add_child_to("Player_Body", body_mesh)` → `override_property("Components/...", "settings", ...)` → add ECS components → `add_child_scene()` for debug label.

### 4c. Player Prefabs (3 core)
- `prefab_player` — inherits `tmpl_character`, adds 5 components, wires `prefab_player_body` as `Body_Mesh`.
- `prefab_player_body` — `Node3D` root + `prefab_character.tscn` child + `MeshInstance3D` direction arrow + `Sprite3D` shadow.
- `prefab_player_ragdoll` — trivial inherit from `tmpl_character_ragdoll.tscn`.
- Live in `scripts/core/editors/` since they are template-reusable.

### 4d. Scene Prefabs (2)
- `prefab_alleyway`, `prefab_bar`
- Import GLB models (`mdl_new_exterior.glb`, `mdl_new_interior.glb`).
- Builder scripts: `create_root("Node3D")` → `add_child_scene(glb_path)` → imperatively add `StaticBody3D` + `CollisionShape3D` with concave collision shape.
- Since GLB data is massive and auto-generated by Godot's import, the builder references the `.glb` directly rather than embedding mesh data.

## 5. Testing Strategy

For each new builder method:
1. Unit test in `test_u_editor_prefab_builder.gd` (RED).
2. Implement method (GREEN).

For each migrated prefab:
1. Builder script under `scripts/demo/editors/` (or `core/` for player).
2. Integration test in `test_u_editor_prefab_builder.gd` that builds, saves, loads, and asserts key node names/component counts.

## 6. Rollback Safety

Each `.tscn` deletion is in its own atomic commit, preceded by builder commit. Original files can be restored via `git checkout HEAD~1 -- path`.
